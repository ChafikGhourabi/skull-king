# Phase 1 — UI Design Contract

> Visual and interaction contract for Phase 1: Foundation.
> Covers auth screens (sign-up, sign-in, email verification, password reset, guest entry,
> guest→account upgrade) and the app shell that all future phases build on.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn/ui (CLI copy-paste, not installed as dependency) |
| Preset | Custom pirate theme — CSS-first Tailwind v4 `@theme` block (see Design Tokens section) |
| Component library | Radix UI primitives (via shadcn CLI) |
| Icon library | lucide-react (latest) |
| Body font | Inter (Google Fonts, weight 400 only) |
| Display / heading font | Cinzel (Google Fonts, weight 700) — serif, Roman letterforms with nautical gravitas; used for page titles and the wordmark only |

**Font rationale:** Cinzel is a classical serif that reads as "ancient, ceremonial, seafaring" without
being an illegible novelty font. Inter covers all body and UI text at high readability on dark
backgrounds. Both are free Google Fonts with good hinting at small sizes.

**shadcn note:** `components.json` does not exist yet. The executor must run `npx shadcn@latest init`
as the first task of Phase 1, then override all CSS variable tokens with the pirate palette below.
Do not use a shadcn preset from ui.shadcn.com — the pirate tokens are defined here and are
incompatible with any preset's neutral palette.

## Spacing Scale

Declared values — all multiples of 4:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, badge padding, chip internal spacing |
| sm | 8px | Input internal padding (vertical), button padding (vertical), stacked form field gaps |
| md | 16px | Default element spacing, input padding (horizontal), section padding (mobile) |
| lg | 24px | Card padding, form section gaps, nav bar padding |
| xl | 32px | Page section gaps, auth card max-width padding |
| 2xl | 48px | Major vertical breaks between form and CTA |
| 3xl | 64px | Page-level top padding on desktop auth screens |

Exceptions:
- Touch targets: minimum 44px height/width for all interactive elements (accessibility requirement — WCAG 2.5.5)
- Nav bar height: 56px (not on 8-point scale — standard app bar height)
- Auth card max-width: 400px on desktop, full-width minus 32px margin on mobile

## Typography

Two fonts. Four sizes. **Two weights: Inter 400 / Cinzel 700.** No exceptions in Phase 1.

| Role | Font | Size | Weight | Line Height | Letter Spacing |
|------|------|------|--------|-------------|----------------|
| Display (page titles) | Cinzel | 32px | 700 | 1.2 | -0.5px |
| Heading (section/card titles) | Cinzel | 20px | 700 | 1.3 | -0.25px |
| Body (form labels, descriptions, button text) | Inter | 16px | 400 | 1.5 | 0 |
| Label / Caption (helper text, error messages, muted info) | Inter | 14px | 400 | 1.4 | 0.1px |

**Weight rule:** Cinzel is display-only at weight 700. All Inter text is weight 400 without
exception — this includes button labels and form field values. Use font-size, letter-spacing,
and spacing differentiation to create visual prominence for interactive elements, not weight.

**Responsive adjustment:** Display size scales to 24px below 480px viewport width. All other
sizes remain fixed.

## Color

60 / 30 / 10 split:

| Role | Token | Approximate value | Usage |
|------|-------|-------------------|-------|
| Dominant (60%) | `--color-background` | `oklch(10% 0.04 35)` — near-black walnut | Page root background, behind all content |
| Secondary (30%) | `--color-surface` + `--color-surface-elevated` | `oklch(15–22% 0.05–0.06 35)` — dark / mid wood | Auth card backgrounds, nav bar, input containers, dialog panels |
| Accent (10%) | `--color-accent` | `oklch(72% 0.20 78)` — pirate gold | See reserved-for list below |
| Destructive | `--color-destructive` | `oklch(55% 0.24 25)` — deep red-orange | Destructive actions only |

**Accent reserved for:**
1. Primary CTA button fill (the main action button on each auth screen)
2. Input focus ring (`outline` + `box-shadow` with `--shadow-gold`)
3. Active nav indicator / underline
4. Link text in body copy (e.g. "Already have an account? Sign in")
5. The Skull King wordmark / logo text
6. Form submission loading spinner

Accent is NOT used for: secondary buttons, icon fills, card borders, non-interactive decorative elements.

**Text on dark backgrounds:** All text is parchment-family (`--color-parchment-100/200/300`).
Never use pure white (#ffffff) — it reads as harsh and breaks the aged-parchment tone.

## Interaction Contracts

### Form Validation Timing

- Field-level validation fires `onBlur` (not `onChange`) to avoid premature errors.
- Exception: password confirmation field validates `onChange` once the user has blurred it once.
- Form-level error banner appears after a failed submission API call, not before.
- All field errors clear when the user begins typing in the errored field (`onChange`).

### Focus Management

- On page load: auto-focus the first input field (email) on all auth screens.
- After successful sign-up: focus moves to the verification screen's "Resend" button.
- After modal/dialog open: focus traps inside (Radix Dialog default behavior — do not override).
- After error: focus moves to the first errored field.

### Sign-Up Flow

```
/signup
  → user submits valid form
  → Supabase createUser called (loading state on button)
  → success → navigate to /verify-email?email={encoded}
  → error → inline alert banner with error copy
```

### Sign-In Flow

```
/signin
  → user submits email + password
  → Supabase signInWithPassword called (loading state)
  → success → navigate to /  (app root — redirected to lobby in Phase 3)
  → error → inline alert banner

/signin → "Continue with Google"
  → Supabase signInWithOAuth({ provider: 'google' }) called
  → browser redirect to Google consent page
  → OAuth callback → /auth/callback route handles token exchange
  → success → navigate to /
  → error → toast: "Could not connect to Google. Please try again."
```

### Email Verification Flow

```
/verify-email
  → display static screen with email address from query param
  → "Resend" button: disabled for 60s after first send (shows countdown: "Resend in 58s")
  → user clicks email link → Supabase verifies token → redirect to /
  → expired link → redirect to /verify-email?error=expired → shows expiry message
```

### Password Reset Flow

```
/forgot-password
  → user enters email → submit
  → Supabase resetPasswordForEmail called
  → always show "Link Sent" success state (do not confirm whether email exists — security)
  → email link → /reset-password?token=... 
  → user enters new password + confirm
  → Supabase updateUser called
  → success → toast "Password updated — welcome back, Captain" + navigate to /signin
  → error → inline alert
```

### Guest Flow

```
/  (or dedicated /guest route)
  → "Play as Guest" → Supabase signInAnonymously called
  → loading state on button
  → success → navigate to / with guest session
  → guest upgrade banner renders persistently below nav bar
  → "Create Account" in banner → navigate to /signup (session preserved, Supabase linkIdentity handles conversion)
  → "Maybe later" → banner dismissed, stored in localStorage key `sk_upgrade_dismissed`
  → Banner does NOT re-appear until next session (per localStorage flag)
```

### Session Persistence

- Supabase client initialized with `persistSession: true` (default) and `autoRefreshToken: true`.
- On app load (`main.tsx` before any route renders): call `supabase.auth.getSession()`.
- If session exists: render app normally (user lands on whatever route they navigated to).
- If no session: redirect unauthenticated routes to `/signin`.
- Protected routes: any route that requires auth redirects to `/signin?redirect={original_path}`.
  After sign-in, user is redirected back to the original path.

### Transitions and Motion

Phase 1 uses minimal motion — the Motion library is installed but card animations belong to Phase 4.

| Transition | Duration | Easing |
|------------|----------|--------|
| Button hover state | 150ms | ease |
| Input focus ring appearance | 100ms | ease |
| Alert banner entrance | 200ms | ease-out (slide down + fade in) |
| Toast entrance | 250ms | ease-out (slide up) |
| Page transition (route change) | none — instant in Phase 1 | — |
| Guest upgrade banner | 300ms | ease-out (slide down from nav bar) |

No animations should be jarring, bouncy, or draw attention to themselves in Phase 1.
The design goal is immersive and calm, not flashy.

## Accessibility Contract

| Requirement | Implementation |
|-------------|---------------|
| Color contrast — body text on surfaces | parchment-100 on wood-900 = ~12.5:1 contrast ratio (AAA) |
| Color contrast — muted text | parchment-300 on wood-900 = ~6.2:1 contrast ratio (AA large text minimum) |
| Color contrast — gold accent on dark | gold-500 on wood-950 = ~5.8:1 (AA for normal text) |
| Color contrast — error text | danger-400 on wood-900 = ~5.1:1 (AA) |
| Minimum touch target | 44px × 44px for all interactive elements |
| Focus visible | All interactive elements have visible focus ring: 2px gold-500 outline, 2px offset |
| Screen reader labels | All icon-only buttons have `aria-label`. Form inputs have associated `<label>` elements (not placeholder-only). |
| Error announcement | Form errors use `aria-live="polite"` region so screen readers announce them |
| Keyboard navigation | Tab order follows visual reading order (top to bottom). No keyboard traps except Radix Dialog (intentional). |
| Reduced motion | Wrap all CSS transitions in `@media (prefers-reduced-motion: no-preference)`. Instant transitions as default. |
