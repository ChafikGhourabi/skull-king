<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → game tables | All mutations blocked by RLS; only SECURITY DEFINER RPCs can write |
| RPC functions → tables | SECURITY DEFINER bypasses RLS; search_path pinned to prevent injection |
| auth.users → profiles | profiles.user_id FK with ON DELETE CASCADE — no orphaned profiles |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-04 | Denial of Service | RLS policies under load | mitigate | All USING clauses use `(select auth.uid())` not bare `auth.uid()` — enables query plan caching per RESEARCH.md Pitfall 6 |
| T-1-07 | Information Disclosure | profiles table | mitigate | RLS enabled on profiles — `(select auth.uid()) = user_id` prevents reading other users' rows |
| T-1-SC | Tampering | supabase CLI install | mitigate | supabase CLI is from official supabase/tap Homebrew tap; verified official source |
| T-DB-01 | Elevation of Privilege | SECURITY DEFINER RPCs | mitigate | `SET search_path = public` on every function prevents search path injection; stubs return not_implemented (no elevation possible with stub logic) |
| T-DB-02 | Tampering | Direct client writes to game tables | mitigate | No INSERT/UPDATE/DELETE RLS policies on game tables — all writes blocked at DB level; only SECURITY DEFINER RPCs can mutate game state |
</threat_model>
