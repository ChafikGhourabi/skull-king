<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| form input → Supabase Auth API | User-entered email/password crosses here; validated client-side then sent to Supabase |
| OAuth redirect → /home | Google consent redirects back with ?code= param; supabase-js auto-exchanges via detectSessionInUrl |
| PASSWORD_RECOVERY event → updateUser | Session must be in PASSWORD_RECOVERY state before updateUser is called |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-03 | Tampering | OAuth PKCE callback | mitigate | Plan 01 set flowType: 'pkce' and detectSessionInUrl: true — supabase-js handles code exchange automatically; no custom callback route needed |
| T-1-05 | Tampering | Guest upgrade via linkIdentity | mitigate | register.tsx uses updateUser for email/password upgrade path (no manual linking required); linkIdentity (OAuth path) is in GuestUpgradeModal (Plan 04) with comment noting Manual Linking dashboard requirement |
| T-AUTH-01 | Information Disclosure | Password reset email existence | mitigate | reset-password.tsx Mode A ALWAYS shows "Link Sent" regardless of API response — do not confirm whether email is registered |
| T-AUTH-02 | Spoofing | Password recovery without valid token | mitigate | Mode B only activates on PASSWORD_RECOVERY onAuthStateChange event — cannot be triggered by URL manipulation; supabase-js validates the recovery token |
</threat_model>
